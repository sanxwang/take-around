最近看到不少 Agent 项目，Skill 列表越写越长：`analyzeCode`、`optimizePerformance`、`fixBug`、`refactorComponent`。看起来能力全面，实际用起来发现：**AI 变成了执行脚本的机器人**。

{/* truncate */}

这篇文章想说清楚：**Skill 的边界在哪，什么时候你其实是在限制模型而不是赋能**。

核心观点：**重 Agent 轻 Skill，不知道怎么合理使用就别用**。

## 什么是 Skill（收敛定义）

### Skill 不是"能力"，是"接口"

别泛讲"赋予 AI 什么能力"，直接给边界：

> **Skill = 模型无法通过推理完成、必须调用外部系统的操作接口**

换句话说：**Skill 是权限，不是智慧**。

### 反例比正例更重要

| 类型 | 示例 | 是否是 Skill | 原因 |
|------|------|-------------|------|
| ❌ 任务 | `analyzeCode` | 否 | 这是思考，不是操作 |
| ❌ 意图 | `fixBug` | 否 | 这是目标，不是手段 |
| ❌ 流程 | `reviewPullRequest` | 否 | 这是组合，不是原子 |
| ✅ 接口 | `readFile(path)` | 是 | 文件系统访问 |
| ✅ 接口 | `applyPatch(diff)` | 是 | 修改文件内容 |
| ✅ 接口 | `runCommand(cmd)` | 是 | Shell 执行 |

### 核心判断标准

**问自己一个问题**：如果把这个 Skill 换成文字描述，模型能理解吗？

- `readFile` 换成 "读取 src/app.ts 的内容" → 能理解，但**没权限**  
  👉 这才是真 Skill

- `fixBug` 换成 "修复这个内存泄漏" → 能理解，也知道怎么做  
  👉 这不是 Skill，是任务委托

**Skill 解决的是权限问题，不是智力问题**。

## Coding Agent 的最小闭环

在讨论什么 Skill 该加、什么不该加之前，先搞清楚一个可用的 Coding Agent 到底需要什么。

### 最小可行循环

```
探索 → 读取 → 修改 → 验证 → 反馈
  ↓       ↓       ↓       ↓       ↓
search  read   edit    run    verify
  ↑_____________________________________↑
              （循环，不是线性）
```

### 三个关键点

1. **`edit` 是核心 Skill** — 决定是否可控
2. **`run/test` 是唯一真反馈** — 没有这个就不是 Agent
3. **其他都是辅助** — search/read 只是上下文采集

### 批评一些产品

很多 AI 编程工具：
- ✅ 会写代码
- ❌ 不会运行验证
- ❌ 不会根据报错调整

**这不叫 Agent，这是高级自动补全**。

判断标准：AI 能否在无人干预下完成一个完整任务并自我验证？

如果不能，它只是个 code generator，不是 agent。

## Skill 设计失控的三种典型模式

这部分是重点。很多团队不是故意把 Skill 设计烂的，而是掉进了这三个坑。

### 模式一：语义过载（把意图当接口）

#### ❌ 错误示例

```yaml
skills:
  - name: fixBug
    description: 修复代码中的 Bug
    parameters:
      - bugDescription: string
      - filePath: string
```

看起来很直观？实际问题很大。

#### 问题分析

**1. 把开放问题变成封闭指令**

用户说："这个组件有内存泄漏"

- 有 `fixBug` skill：
  ```
  AI: 调用 fixBug(bugDescription="内存泄漏", filePath="App.tsx")
  系统: 返回修改建议
  AI: 应用修改
  ```
  
  👉 **AI 失去了探索空间**

- 没有这个 skill：
  ```
  AI: 先 readFile("App.tsx")
  AI: 发现用了 useEffect，search "useEffect cleanup"
  AI: 分析依赖数组
  AI: readFile 导入的 hook 代码
  AI: 找到问题根源
  AI: edit + 解释原因
  ```
  
  👉 **AI 会思考，会探索**

**2. 本质是在用 Skill 名称做 prompt engineering**

如果你发现自己在 Skill 的 `description` 里写了很长的逻辑说明，那这个 Skill 多半是多余的。

#### 真实影响

一个对比实验（真实案例简化）：

| 场景 | 有 `fixBug` skill | 无（只有 read/edit/run） |
|------|------------------|-------------------------|
| 解决简单问题 | ✅ 快 | ✅ 稍慢但更透明 |
| 解决复杂问题 | ❌ 拿到空结果后卡住 | ✅ 会多次探索 |
| 错误率 | 高（一次性修改） | 低（循环验证） |

**结论**：Skill 越语义化，模型越不会思考。

---

### 模式二：粒度爆炸（过度细分）

#### ❌ 错误示例

```yaml
skills:
  - readLine
  - readBlock  
  - readFunction
  - readClass
  - readImport
  - readExport
  - readType
  - readComment
  - readDocstring
```

初衷可能是"提供更精准的读取能力"，实际效果是灾难。

#### 问题分析

**1. 调用成本 > 推理成本**

模型需要：
- 理解 9 个接口的区别
- 选择用哪个（决策树复杂度 O(n²)）
- 记住每个的返回格式

还不如直接：
```javascript
readFile(path, options?: { range?: [start, end], symbolName?: string })
```

**2. Agent 变成 API 调度器**

大量 token 浪费在"我应该用 readFunction 还是 readClass"。

#### 对比测试

同样的任务："找到 UserProfile 组件的 props 定义"

- **粗粒度**（1 个 skill）：
  ```
  readFile("UserProfile.tsx") → 一次调用，模型自己找 props
  ```

- **细粒度**（9 个 skill）：
  ```
  我应该用哪个？readType? readClass? readFunction?
  → 尝试 readType → 没找到
  → 尝试 readFunction → 找到了
  → 但多花了 3 次调用
  ```

**规律**：**推理能力强的模型，更适合粗粒度工具**。

#### 什么时候该细分？

只有一个理由：**性能优化**。

如果发现模型总是读取大文件但只需要一小块，再提供细粒度选项。

但默认应该是粗粒度。

---

### 模式三：流程硬编码（隐式锁死决策）

这是最隐蔽的一种。表面上 Skill 定义合理，但使用方式被固定了。

#### ❌ 错误示例

系统文档里写：

```markdown
## Coding Workflow

1. 使用 `search` 找到相关文件
2. 使用 `read` 读取内容
3. 使用 `edit` 修改代码
4. 结束
```

看起来像"最佳实践"，实际是把 Agent 变成脚本执行器。

#### 问题分析

**现实应该是循环，不是线性**：

```
read ↔ search ↔ analyze
  ↓
edit
  ↓
run → 失败？ → 重新 read/search/analyze
  ↓
verify → 成功
```

#### 真实案例

某个 Agent 修改了一个 React 组件：

1. ✅ 读了 `Button.tsx`
2. ✅ 修改了代码（添加了新 prop）
3. ❌ **没读 `Button.test.tsx`**
4. ❌ 修改后测试挂了（prop 类型不匹配）
5. ❌ **没有 run test 验证**
6. ❌ **无法回溯**（流程已经"完成"）

用户收到的是一个看起来正确、但跑不起来的代码。

#### 如何避免

**不要在文档里写"标准流程"，而是给"反馈信号"**：

```yaml
# ❌ 不要这样写
workflow:
  - step1: search
  - step2: read
  - step3: edit

# ✅ 应该这样写
evaluation:
  success: tests pass + no errors
  failure: re-analyze and retry
```

让 Agent 自己决定要走几个循环。

## 合理的 Skill 设计原则

讲完反例，给正面建议。

### 原则 1：只暴露物理能力，不暴露认知能力

| 类型 | 示例 | 判断 |
|------|------|------|
| 物理能力 | 文件系统、Shell、网络请求、数据库 | ✅ 应该做成 Skill |
| 认知能力 | 分析代码、优化性能、设计架构 | ❌ 这是模型的本职工作 |

**关键区分**：
- 物理能力：**人类也需要工具**
- 认知能力：**这是模型存在的意义**

#### 实例

```javascript
// ✅ 物理能力
const skills = {
  readFile: (path) => fs.readFileSync(path),
  writeFile: (path, content) => fs.writeFileSync(path, content),
  runCommand: (cmd) => execSync(cmd),
  httpGet: (url) => fetch(url)
};

// ❌ 认知能力（别做成 skill）
const notSkills = {
  analyzeComplexity: (code) => ..., // 让模型自己分析
  suggestRefactor: (code) => ...,   // 让模型自己思考
  reviewCode: (code) => ...         // 这是任务，不是工具
};
```

---

### 原则 2：默认大颗粒，按失败再细化

#### ✅ 渐进式设计

**第一版**：
```typescript
readFile(path: string): string
```

**发现问题**：模型经常只需要部分内容，读全文件浪费 token

**第二版**：
```typescript
readFile(path: string, options?: { 
  lineRange?: [number, number] 
}): string
```

**再发现问题**：有时只需要某个函数

**第三版**：
```typescript
readFile(path: string, options?: { 
  lineRange?: [number, number],
  symbolName?: string  // 读取特定函数/类
}): string
```

#### ❌ 一开始就拆碎

直接提供 10 种读取方式：
```typescript
readFileFull()
readFileLines()
readFileRange()
readFileFunction()
readFileClass()
readFileImports()
...
```

模型不知道选哪个，反而降低效率。

#### 判断标准

**只有当发现明确的性能瓶颈时，才细化粒度**。

---

### 原则 3：所有写操作必须可回滚（diff-based）

这是安全底线。

#### ✅ 安全的 edit

```typescript
applyPatch(filePath: string, diff: string): {
  success: boolean,
  preview?: string,
  error?: string
}
```

**好处**：
1. 可 preview（用户能看到要改什么）
2. 可 rollback（改错了能撤销）
3. 可 version control（能看历史）
4. 粒度可控（只改必要部分）

#### ❌ 危险的 edit

```typescript
overwriteFile(filePath: string, content: string): void
```

**问题**：
- 无法回溯
- 无法 preview
- 容易丢失代码
- **Agent ≈ 破坏脚本**

#### 真实教训

某个 Agent 系统用了 `overwriteFile`：

```
用户: 帮我优化这个文件
AI: 好的（读取文件）
AI: （生成优化后的代码）
AI: overwriteFile("app.tsx", optimizedCode)
系统: ✅ 完成
用户: 卧槽我的注释全没了
```

原因：模型生成代码时丢了注释。

如果用 `applyPatch`：
- 用户能在 preview 里看到注释被删了
- 可以阻止应用
- 或者让 AI 重新生成保留注释的版本

---

### 原则 4：必须存在真实反馈通道

#### ❌ 没有 `run/test` 的后果

```
AI 修改代码 
  → 用户手动测试 
  → 发现问题 
  → 告诉 AI 
  → 再修改
  → 用户再测试
  → ...
```

**这不是 Agent，是人肉循环**。

#### ✅ 真正的 Agent

```
AI 修改代码 
  → AI 自己 run test 
  → 看到报错 
  → 分析错误 
  → 自己调整
  → 再次 run test
  → 通过 ✅
```

#### 实现方式

```typescript
const skills = {
  edit: applyPatch,
  verify: {
    runTests: (pattern?: string) => execSync(`npm test ${pattern}`),
    lint: () => execSync('npm run lint'),
    typeCheck: () => execSync('tsc --noEmit'),
    build: () => execSync('npm run build')
  }
};
```

**判断标准**：AI 能否在无人干预下完成一个完整任务？

如果不能，你做的是 code generator，不是 agent。

## 一个更现实的 Skill 清单

讲了这么多原则，给一个实际的参考清单。

### Coding Agent 的核心 Skill（5-8 个足够）

```typescript
// 1. 文件系统
readFile(path: string, options?: ReadOptions): string
writeFile(path: string, diff: string): Result  // diff-based

// 2. 代码搜索
search(pattern: string, options?: SearchOptions): SearchResult[]

// 3. 执行验证
runCommand(command: string, options?: ExecOptions): ExecResult
runTests(pattern?: string): TestResult

// 4. 项目理解
listFiles(pattern?: string): string[]
getProjectStructure(): Tree

// 5. 版本控制（可选）
gitDiff(): string
gitStatus(): Status
```

**就这些**。

### 不需要的 Skill

```typescript
// ❌ 这些都不需要
analyzeCode()      // 让模型自己分析
optimizePerformance()  // 让模型自己优化
fixBug()           // 让模型自己修
generateTests()    // 让模型自己写
reviewCode()       // 让模型自己 review
refactorComponent() // 让模型自己重构
```

**如果一个 Skill 可以用"让模型写 prompt 给自己"实现，那就不该是 Skill**。

## 一个对比实验

我们实际测试了两个 Agent 配置：

### 配置 A：丰富 Skill（15 个）

```yaml
skills:
  - readFile, readFunction, readClass, readImport
  - editFile, editFunction, insertCode
  - search, searchSymbol, searchUsage
  - analyzeCode, detectBug
  - runTests, runLint, typeCheck
```

### 配置 B：精简 Skill（5 个）

```yaml
skills:
  - readFile (支持 range/symbol 参数)
  - applyPatch (diff-based)
  - search (支持 pattern/scope 参数)
  - runCommand
  - listFiles
```

### 测试任务：修复一个真实 Bug

**任务**："修复 UserProfile 组件的内存泄漏"

**结果**：

| 指标 | 配置 A（15 skill） | 配置 B（5 skill） |
|------|-------------------|------------------|
| 平均 Skill 调用次数 | 12 次 | 8 次 |
| 首次修复成功率 | 45% | 68% |
| 平均修复时间 | 2.3 分钟 | 1.8 分钟 |
| 无效调用率 | 33% | 12% |

**发现**：
1. Skill 多不等于能力强
2. 粗粒度 Skill 反而提升成功率
3. 模型在选择工具上浪费了大量推理

## 总结

### 三句话记住本文

1. **Skill 是接口，不是能力** — 别把任务包装成工具
2. **粗粒度 > 细粒度** — 推理能力强的模型需要自由度  
3. **必须有反馈循环** — 没有 run/test 就不是 Agent

### 设计 Skill 的三个问题

在添加任何 Skill 之前，先问自己：

#### 问题 1：这个 Skill 模型能推理出来吗？

- 能 → **别加**，浪费
- 不能 → 继续问题 2

#### 问题 2：这个 Skill 会限制模型的决策空间吗？

- 会（如 `fixBug`、`optimizeCode`）→ **别加**，有害
- 不会 → 继续问题 3

#### 问题 3：有真实反馈通道吗？

- 有（能 run/test）→ ✅ **可以加**
- 没有 → 先加反馈能力

### 记住一句话

> **Skill 的本质是"给 AI 手"，  
> 但一旦你开始规定"手该怎么用"，  
> 你就在悄悄替它做决定。**

## 行动建议

如果你正在设计 Agent：

**第一步**：列出当前所有 Skill

**第二步**：用三个问题过滤
- [ ] 这是物理能力还是认知能力？（认知能力删掉）
- [ ] 粒度是否太细？（能合并就合并）
- [ ] 有没有写操作的回滚机制？（没有就加上）

**第三步**：确保反馈闭环
- [ ] 能 run code 吗？
- [ ] 能 run tests 吗？
- [ ] 能看到错误信息吗？

**第四步**：观察使用模式
- Skill 调用是否过于集中？（说明其他 Skill 多余）
- 是否有大量无效调用？（说明粒度不对）
- 是否总是线性执行？（说明缺少反馈）

**最重要的原则**：

> **重 Agent 轻 Skill**  
> **不知道怎么合理使用，就别用**
