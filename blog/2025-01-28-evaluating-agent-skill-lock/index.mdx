{/* truncate */}

最可怕的是，这个过程完全看不见。表面成功率可能很高，但 Agent 的推理能力正在悄悄死亡。本文给出四个可量化的指标，帮你诊断。

## 问题可能已经出现了，只是你还没发现

### 一个隐形的危机

你接手了一个 Coding Agent 项目，运行了几次演示：

- ✅ 能修改代码
- ✅ 能运行测试
- ✅ 成功率看起来不错

但逐渐你会发现：

- ❌ 只会用同样的方法解决问题
- ❌ 遇到新场景就容易失败
- ❌ 错误发生后，需要人工干预才能调整
- ❌ 有时候做得越来越差（而不是越来越好）
- ❌ 改动总是集中在固定的几个地方

**这不是 Agent 变弱了，而是被 Skill 锁死了**。

### 为什么表面上看不出问题

因为 Skill 的伤害是**隐性的、逐步的、统计性的**：

1. **短期收益掩盖长期成本**
   - 初期：Skill 确实提升了成功率 (+15%)
   - 中期：但探索能力开始下降
   - 长期：最终失败率反而上升

2. **成功的任务看不出问题**
   - 对于"标准型"问题：Skill 表现很好
   - 但你无法判断是 Skill 强，还是问题太简单

3. **失败的原因被隐藏了**
   - 调试日志显示："fixBug Skill 执行失败"
   - 你看不到：其实是"限制了 Agent 的思考"
   - 真实原因：这个 Bug 需要跨越 Skill 的边界才能理解

### 四个指标能告诉你什么

这四个指标就是"症状"，从不同角度诊断同一个病：**Skill 正在替代思考**。

## 指标 1：Skill 使用熵 — 固定的套路

衡量 Agent 是在"灵活思考"还是"重复同一套路"。

### 问题的表现

你发现 Agent 在处理不同任务时，**行为完全相同**：

```
任务 A: 修复内存泄漏
  1. search useEffect  ← 总是这样
  2. read file
  3. edit dependencies
  4. done

任务 B: 修复类型错误
  1. search useEffect  ← 仍然是这样！
  2. read file
  3. edit type definition
  4. done

任务 C: 优化性能
  1. search useEffect  ← 还是这样！
  2. read file
  3. edit render logic
  4. done
```

**这就是信息论上的"低熵"** — 行为可预测，缺乏灵活性。

### 为什么会这样

因为 Agent 被 Skill 引导了。它学到的不是"怎么思考"，而是"哪些 Skill 管用"：

```
Skill useEffectFix 很常用？
  → 继续用它

需要处理其他问题？
  → 还是用 useEffectFix
  
不行？
  → 试试其他 Skill
  
还是不行？
  → 放弃
```

**而不是**：

```
分析这个问题的本质是什么？
尝试不同的方法
验证哪个有效
调整策略
```

### 计算和诊断

```javascript
// 收集 100 次任务中的 Skill 调用分布
const skillUsage = {
  readFile: 85,
  search: 10,
  edit: 5
};

// 计算熵
function calculateEntropy(usage) {
  const total = Object.values(usage).reduce((a, b) => a + b, 0);
  const probs = Object.values(usage).map(count => count / total);
  return -probs.reduce((sum, p) => {
    return sum + (p > 0 ? p * Math.log2(p) : 0);
  }, 0);
}

const entropy = calculateEntropy(skillUsage);
// 结果: 0.3 (很低)
```

### 诊断标准

| 熵值 | 含义 | 危险信号 |
|-----|------|--------|
| < 0.5 | **极低** | ⚠️ 完全被锁死，行为机械化 |
| 0.5-1.0 | **偏低** | ⚠️ 有问题，只用固定几个 Skill |
| 1.0-1.5 | **健康** | ✅ 根据情况灵活调整 |
| > 1.5 | **过高** | ⚠️ 可能是乱调用，需要检查 |

### 最让人担心的场景

```javascript
// 100 次任务中的熵值
month1: 0.95  // 还不错
month2: 0.87  // 开始下降
month3: 0.73  // 明显下降
month4: 0.52  // 被锁死了
```

**表面理由**："系统越来越稳定"

**真实情况**："Agent 放弃思考了"

## 指标 2：无效调用率 — 被迫做多余的工作

衡量 Skill 有多少是"用来试错"而不是"真正解决问题"。

### 问题的表现

你在日志里看到这样的场景：

```
1. readFile('Button.tsx')
2. readFile('Button.test.tsx')
3. readFile('Button.types.ts')
4. edit Button.tsx
   最后只修改了 Button.tsx，前两个文件没用上
   → 60% 的读取是浪费的
```

或者：

```
1. search('useState hooks')
2. search('useEffect cleanup')
3. search('memory leak')
4. read 第二个搜索结果
   最后只用了第一个
   → 三分之二的搜索调用浪费了
```

**这叫"无效调用"**。

### 为什么会产生无效调用

**根本原因**：Skill 太多、太细，Agent 需要反复试错才能找到对的

```
有 10 个读取 Skill：
  readLine / readBlock / readFunction / readClass...

Agent 的决策树：
  我应该用哪个？ → 试试 readFunction → 失败
  那试试 readClass？ → 成功了

vs

只有 1 个读取 Skill：
  readFile(path, {range, symbol})

Agent 的决策：
  直接读，一次成功
```

**Skill 越细分，Agent 需要越多的"试错调用"来找到正确的**。

### 计算

```javascript
class SkillTracker {
  constructor() {
    this.calls = [];
  }
  
  recordCall(skillName, result) {
    this.calls.push({
      skill: skillName,
      result,
      used: false
    });
    return this.calls.length - 1;
  }
  
  markUsed(callId) {
    if (this.calls[callId]) {
      this.calls[callId].used = true;
    }
  }
  
  getInvalidRate() {
    const unused = this.calls.filter(c => !c.used).length;
    return unused / this.calls.length;
  }
}
```

### 诊断标准

| 无效率 | 含义 | 问题严重程度 |
|------|------|-----------|
| < 15% | 健康 | ✅ Skill 粒度合理 |
| 15% - 30% | 中等 | ⚠️ 有一些浪费，可以优化 |
| 30% - 50% | 严重 | 🚨 Agent 在浪费 token，成本很高 |
| > 50% | 极度浪费 | 🚨 超过一半的调用都没用，系统有严重问题 |

### 真实案例

#### 案例 A：无效率 15%（健康）

```
100 次 Skill 调用：
  readFile 80 次 → 80 次全用上
  edit 10 次 → 8 次用上，2 次被后续覆盖
  search 10 次 → 9 次用上，1 次搜错了
  
无效率 = 3/100 = 3% ✅ 很好
```

#### 案例 B：无效率 42%（严重问题）

```
100 次 Skill 调用的分布：
  readLine 20 次 → 只有 8 次用上
  readBlock 18 次 → 只有 7 次用上
  readFunction 15 次 → 只有 6 次用上
  readClass 12 次 → 只有 3 次用上
  edit 20 次 → 20 次用上
  search 15 次 → 12 次用上

无效率 = 42/100 = 42% 🚨
```

**为什么？** Skill 太细分了，Agent 不知道该用哪个，反复试错。

### 这个指标的真正含义

**无效率 = "Agent 在用 Skill 清单猜测，而不是思考"的成本**

- 无效率高 → Skill 设计导致了不必要的决策开销
- 高成本 → 同样的功能，浪费了更多 token
- 更重要的 → Agent 的推理被"选择 Skill"这个任务占据了

**所以，无效率高不仅是浪费，更是一个信号：Agent 正在"被工具玩"**

## 指标 3：回溯深度 — 放弃思考的信号

衡量 Agent 是"脚本执行"还是"真正的问题解决"。

### 问题的表现

你看到 Agent 的行为模式变了：

```
之前：
  read → analyze → edit → test → 测试失败 
    ↓ 发现问题
  re-read → re-analyze → edit → test → 成功

现在：
  read → edit → test → 失败
  用户干预说："再试一下"
  重新开始...
```

**能力没变，思考过程消失了**。

### 为什么会这样

当引入了很多"做决定"的 Skill 后，Agent 停止了自我修正：

```
有了 fixBug Skill：
  调用 fixBug(description) → 返回结果
  测试失败 → 我不知道为什么失败（黑盒）
  → 无法回溯，无法调整

vs

只有基础 Skill：
  read → analyze → edit → test
  测试失败 → 我看到了报错信息
  → 可以重新 read → 分析新的错误
  → 循环直到成功
```

### 定义和计算

**回溯 = 任务过程中，返回上一步重新决策的次数**

```javascript
class BacktrackingTracker {
  constructor() {
    this.steps = [];
    this.backtracks = 0;
  }
  
  pushStep(action) {
    this.steps.push(action);
  }
  
  backtrack() {
    // 返回上一步，重新尝试不同的方法
    this.backtracks++;
    this.steps.pop();
  }
  
  getBacktrackingDepth() {
    return this.backtracks;
  }
}
```

### 诊断标准

| 回溯深度 | 含义 | 严重程度 |
|---------|------|--------|
| 0 | **完全线性** | 🚨 被完全锁死，一条路走到黑 |
| 1 | **几乎没有** | 🚨 有反馈但不会反思 |
| 2-3 | **基本有效** | ✅ 能自我修正 |
| 4-6 | **健康** | ✅ 真正在问题解决 |
| > 8 | **陷入循环** | ⚠️ 可能卡住了，需要优化 |

### 真实案例对比

#### 案例 A：回溯深度 = 0（被锁死）

```
1. search('useState memory leak') 
   ← 搜索了，但基于第一个结果就确定了
2. read('hooks.ts')
   ← 读了，但没再读别的相关文件
3. edit (添加 cleanup)
4. runTests()
   ← 失败：但这里就结束了
   
最终：用户手动看错误，告诉 AI 问题在哪
```

**特点**：Agent 不会自己发现问题。

#### 案例 B：回溯深度 = 3（正常工作）

```
1. search('useState hooks')
2. read('app.tsx')
3. analyze：useEffect 有问题
4. edit (添加 cleanup)
5. runTests() → 失败：还有引用问题
   
   ↓ 回溯 1：发现问题不只在 cleanup
   
6. read('hooks.ts')  ← 重新读相关文件
7. analyze：引用也需要处理
8. edit (修改引用)
9. runTests() → 失败：还有类型问题
   
   ↓ 回溯 2：继续深挖
   
10. read('types.ts')  ← 再读相关文件
11. analyze：类型定义不匹配
12. edit (修改类型)
13. runTests() → 成功
    
    ↓ 回溯 3 次才成功
```

**特点**：每次失败后，Agent 都会重新分析，逐步逼近真实问题。

### 最可怕的数据

```
单个任务的回溯深度变化：
任务 1: 深度 3 (好的状态)
任务 2: 深度 2
任务 3: 深度 1
任务 4: 深度 0  ← 完全不回溯了
任务 5-10: 深度 0 (固定了)
```

**这说明什么？**

- Agent 的 Skill 正在强化某种固定行为
- 它学会了"最常见的解法"
- 但丧失了"遇到新问题时的适应能力"
- 最终变成了一个**模板执行器**

### 这个指标的真正意义

**回溯深度 = Agent 还有多少"反思能力"**

- 深度 0：没有反思，完全是脚本
- 深度 2-3：还能自我纠正，有点智能
- 深度 4+：真正在"问题解决"

**一旦深度降到 0，你拥有的不是 Agent，而是一个确定性的程序**。

## 指标 4：编辑成功率 — 信心过度导致的错误

衡量一次修改就能正确的比例，背后反映的是上下文污染程度。

### 问题的表现

你发现一个模式：

```
AI 生成的 patch 经常第一次就成功 → 看起来很强

但继续看会发现：
  - 引入了新的 bug
  - 把注释删了
  - 把其他依赖改错了
  - 只有测试通过了，但实际逻辑有问题
```

**表面上看成功率很高，实际上是在制造隐形问题**。

### 为什么会这样

当 Skill 太多时，上下文被"污染"了：

```
AI 的脑子里：
  - 各种 Skill 的定义
  - 使用 Skill 的规则
  - 处理失败的流程
  - 回滚机制
  - ... （还有很多）

实际用于理解任务的上下文空间：
  可能只剩下 50%
```

结果：
- ❌ 信息不足，导致错误决策
- ❌ 或者太自信（因为有了 Skill），忽视细节
- ❌ 或者试图"一步到位"，不经过多步验证

### 诊断标准

| 成功率 | 含义 | 问题分析 |
|------|------|--------|
| > 85% | **过高** | ⚠️ 可能是任务太简单，或者 Agent 在冒险 |
| 60-70% | **健康** | ✅ 需要反复验证，说明在真正问题解决 |
| 40-60% | **偏低** | 🚨 上下文或 Skill 有问题 |
| < 40% | **很低** | 🚨 系统有严重问题 |

### 真实案例对比

#### 案例 A：高成功率 88% 的陷阱

```
看起来：
  100 次编辑，88 次一次成功 ✅

但仔细看：
  - 27 次虽然测试通过，但引入了新警告
  - 12 次删了代码注释
  - 8 次改动范围太广，影响了其他模块
  - 18 次虽然没报错，但实际有隐形 bug

真实分析：
  - 完全正确的修改：23 次 (23%)
  - 需要反馈的修改：65 次 (65%)
  - 完全错误：12 次 (12%)
```

**表面成功率 88%，实际只有 23% 是真的成功**。

#### 案例 B：成功率 65% 的真实情况

```
100 次编辑：
  - 65 次测试通过
  - 35 次测试失败，需要重试

但重试时：
  - 30 次在第二轮成功
  - 4 次需要第三轮
  - 1 次无法解决

总体：99% 的任务最终成功

和案例 A 相比：
  虽然"一次成功率"低，但"最终成功率"反而更高
  而且没有隐形 bug
```

### 这个指标的隐藏含义

**首次成功率低 = Agent 在谨慎思考**

```
低成功率的好处：
  ✅ 有反馈循环
  ✅ 能检测问题
  ✅ 有修复机制
  ✅ 最终更稳定

高成功率的问题：
  ❌ 可能漏掉问题
  ❌ 无法自我修正
  ❌ 隐性 bug 存在
  ❌ 看起来强，实际脆弱
```

**在 Skill 滥用的系统里，高成功率反而是个危险信号**。

## 总结：四个指标速查表

如果你现在就想诊断你的 Agent，看这个表：

| 指标 | 被锁死的表现 | 健康的表现 |
|------|-----------|--------|
| 使用熵 | < 0.5（固定套路） | 0.8 - 1.5（灵活调整） |
| 无效调用率 | > 40%（浪费能力） | < 20%（精准调用） |
| 回溯深度 | 0（脚本执行） | 2 - 4（自我修正） |
| 首次成功率 | > 85%（虚假强大） | 50 - 70%（真实能力） |

**如果你的 Agent 看起来都不对，那就是被锁死了**。

## 核心认知转变

这篇文章的真实目的不是"教你怎么评估"，而是让你看到：

### 1. Skill 越多，不一定越强

- 多了，Agent 需要花精力"选择 Skill"
- 选错了，整个方向都偏了
- 最后变成：大部分能力消耗在"决定用哪个工具"上，而不是"解决问题"上

### 2. 表面成功率高可能是个坏信号

- 说明问题太简单，或者 Agent 在冒险
- 没有反馈循环意味着隐形问题无法发现
- 短期看起来强，长期会出问题

### 3. 被锁死的过程很隐蔽

- 不会突然失效
- 而是逐步机械化：熵下降、无效率上升、回溯消失
- 等你发现时，已经很难救回来了

### 4. 你需要的是克制，不是扩展

不是"加更多 Skill 来适应更多场景"，而是"减少 Skill，让 Agent 有思考空间"。

---

## 最重要的警告

**大多数失败的 Agent 项目，死因不是"能力不够"，而是"被优化成了脚本执行器"。**

表面看起来：
- ✅ 系统稳定
- ✅ 成功率高
- ✅ 自动化完成

实际发生的：
- ❌ Agent 放弃思考
- ❌ Skill 变成了枷锁
- ❌ 遇到新场景就崩溃

而且这个过程完全看不见，直到你拉出这四个指标来看。

---

## 下一步：如果你的指标已经很差

不要加更多 Skill，而是：

### 第一步：恢复反馈循环
确保 Agent 能 `run/test`，能看到真实的失败信息。

### 第二步：合并冗余 Skill
把 readLine / readBlock / readFunction 这种细分的东西合并回 readFile。

### 第三步：让模型重新思考
不要用"指导 Skill"（如 fixBug、optimize），只提供"操作 Skill"（read、edit、run）。

### 第四步：重新测量
四个指标应该慢慢恢复：
- 熵值上升（灵活性回来了）
- 无效率下降（决策精准了）
- 回溯深度上升（能反思了）
- 成功率适度（真实能力显现了）

真正的问题是："我已经给了太多 Skill，为什么系统反而变差了？"
